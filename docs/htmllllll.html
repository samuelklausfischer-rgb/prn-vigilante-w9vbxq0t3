⚠️ Importante: Migrations não funcionam via Edge Function

O Supabase NÃO PERMITE executar DDL (CREATE TABLE, ALTER TABLE, etc.) via Edge Functions.

As migrations precisam ser executadas manualmente pelo Supabase Dashboard:

  • Acessar https://app.supabase.com/project/.../sql
  • Clicar em "New Query"
  • Colar o SQL
  • Executar as migrations nesta ordem:
    1. 1. Criar 4 novas tabelas (message_logs, worker_heartbeats, patient_consent, message_blocks)
    2. 2. Adicionar 4 campos a patients_queue (locked_by, locked_at, attempt_count, last_message_at)
    3. 3. Adicionar 2 campos a whatsapp_instances (last_message_at, messages_sent_count)
    4. 4. Criar 6 índices para performance
    5. 5. Criar 2 funções SQL (claim_next_message, release_expired_locks)
    6. 6. Criar 2 views (dashboard_realtime_metrics, expired_locks)
  • Validar que as tabelas/índices/funções foram criadas corretamente antes de continuar
  • Se der erro, anotar e criar plano de rollback
📋 Como Voltar à Automação

Executar as 15 operações SQL listadas acima no Supabase Dashboard manualmente.

3
Implementar Drivers e Serviços
Criar estrutura de pastas em /automation/src/ e implementar:
  • Supabase Driver (claim_next_message, release_expired_locks)
  • Evolution Driver (sendMessage, getSafeDelay)
  • Queue Service (claim, updateStatus, markAsDelivered)
  • Instance Service (selectInstance, updateMetrics)
  • Worker Engine (motor principal de automação)
4
Implementar Features Avançadas
Resiliência (Retry, Circuit Breaker)
Humanização (Delays 3-6-12 minutos, Spinning 7-10s, Split de mensagens)
Métricas e Logs (Dashboard em tempo real, alertas, tracing)
📝 Recomendações Antes de Começar
  • Fazer backup do banco de dados antes de executar migrations
  • Testar migrations em ambiente de desenvolvimento antes da produção
  • Manter plano de rollback documentado (COMMIT/ROLLBACK)
  • Validar que as funções SQL estão criadas corretamente antes de usar no código
  • Ter as credenciais da Evolution API configuradas para o Evolution Driver
