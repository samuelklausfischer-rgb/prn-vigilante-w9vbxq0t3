================================================================================
PARTE 3 - PONTO EXATO ONDE O BUG PODE ESTAR
================================================================================

BUG IDENTIFICADO: INCREMENTO PREMATURO DE 'processed'

Localizacao Precisa:
- Arquivo: automation/src/core/worker-engine.ts
- Linha: 146
- Codigo: this.processed += 1

Cronograma Real de Eventos:
T1: sendTextMessage() -> Evolution API retorna sucesso
T2: markMessageAccepted() -> status='sending'
T3: syncDeliveryStatus() inicia polling (assincrono)
T4: [BUG] this.processed += 1 executado IMEDIATAMENTE X
T5+: Polling tenta confirmar entrega, mas ja foi "processado"

================================================================================
4 EVIDENCIAS DO BUG
================================================================================

Evidencia 1: Discrepancia entre incrementos e atualizacoes SQL
- Worker incrementa this.processed (linha 146)
- Mas NAO chama markMessageDelivered()!
- markMessageDelivered() so e chamado em DRY RUN mode (linha 272)
- No modo normal, JAMAIS e chamado! X

Evidencia 2: markMessageAccepted() vs markMessageDelivered()
- markMessageAccepted(): marca status='sending' X
- markMessageDelivered(): marca status='delivered'
- markMessageAccepted() NAO atualiza delivered_at
- markMessageDelivered() NAO e chamado no modo normal X

Evidencia 3: syncDeliveryStatus() e assincrono e pode falhar
- Iniciado com .catch() -> erros silenciados
- Tenta 3 vezes: 60s, 300s, 900s
- Se todas falharem: WARNING e worker continua
- Mensagem fica com status='sending' para sempre X

Evidencia 4: status='sending' vs last_delivery_status
- syncDeliveryStatus() atualiza delivered_at e last_delivery_status
- MAS NAO atualiza status para 'delivered' X
- Resultado: status permanece 'sending' mesmo apos polling

================================================================================
IMPACTO DO BUG
================================================================================

1. Inconsistencia Worker vs Banco:
   - Worker.processed: X (inclui mensagens ainda nao entregues)
   - Banco (status='delivered'): Y < X (apenas entregues confirmadas)

2. Dashboard incorreto:
   - Mostra processed=X mas entregues=Y
   - Usuario ve X entregues mas apenas Y foram entregues

3. Mensagens travadas:
   - Se polling falhar, status='sending' para sempre
   - Worker ja considerou como "processada"

4. Follow-ups incorretos:
   - runSecondCallRecovery() verifica delivered_at
   - Se polling falhou, mensagem e "nao recebida"
   - Pode gerar follow-ups desnecessarios
